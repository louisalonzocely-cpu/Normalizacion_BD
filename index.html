Contexto

¿Por qué normalizar?

Este proceso es fundamental en bases de datos transaccionales donde las actualizaciones son frecuentes, equilibrando la eficiencia de las escrituras con la consistencia de la información.

¿Qué es la redundancia de datos?

La redundancia de datos en una base de datos se refiere al almacenamiento repetido de la misma información en diferentes lugares dentro del sistema.

Cómo lo soluciona

La normalización aplica el principio de una sola fuente de verdad: cada dato se almacena en una única tabla y se relaciona mediante claves foráneas, eliminando duplicados y garantizando consistencia.

¿Qué son las anomalías de BD?

Inconsistencias o errores que surgen durante las operaciones de actualización, inserción o eliminación, generalmente debido a un esquema mal diseñado o fallas de integridad.

Cómo lo soluciona

Al separar los datos en tablas por entidad (1FN, 2FN, 3FN), cada operación afecta un solo lugar: puedes insertar un cliente sin ventas, actualizar un dato una vez sin inconsistencias, y eliminar un pedido sin perder información del cliente.

¿Qué es la integridad referencial?

Es un conjunto de reglas en las bases de datos relacionales que garantiza que las relaciones entre tablas permanezcan consistentes y válidas, asegurando que las claves foráneas apunten únicamente a claves primarias existentes o sean nulas.

Cómo lo soluciona

El modelo 3FN combinado con restricciones FOREIGN KEY en el SGBD garantiza que toda relación entre tablas sea consistente: no puedes crear un pedido sin un cliente existente ni borrar un cliente que tenga pedidos asociados.

Definición formal

Es un proceso de diseño que organiza la información en tablas específicas para reducir la redundancia y mejorar la integridad de los datos. Este método divide tablas grandes en otras más pequeñas y establece relaciones mediante claves, lo que permite eliminar inconsistencias y anomalías en las actualizaciones.
El objetivo principal es asegurar que cada dato se almacene una sola vez, facilitando así el mantenimiento, la eficiencia de las consultas y el crecimiento del sistema sin duplicaciones innecesarias.

Tipos de anomalías

Anomalía de Inserción

Ocurren cuando no es posible añadir datos a la base de datos porque faltan valores requeridos por otras columnas (por ejemplo, no poder registrar un autor si no tiene libros asociados).

Anomalía de Actualización

Surgen cuando la información redundante debe actualizarse en múltiples registros; si se modifica solo una instancia, se generan inconsistencias en los datos.

Anomalía de Eliminación

Se producen al borrar un registro, lo que provoca la pérdida involuntaria de información vital sobre otras entidades relacionadas (por ejemplo, perder datos de un estudiante al eliminar su última clase).

Caso práctico

El caso: BrunosFruit Cartagena

BrunosFruit es una distribuidora de frutas y verduras ubicada en el Mercado de Bazurto, Cartagena. Lleva sus pedidos en un cuaderno con una tabla que mezcla clientes, productos y rutas de reparto en un solo registro. Al no tener una base de datos normalizada, la misma información se repite varias veces, generando inconsistencias y dificultando las consultas.

Problema 1FN detectado

productos y cantidades almacenan múltiples valores separados por coma en una sola celda. Esto viola la atomicidad de la 1FN: no podemos consultar ni actualizar un producto individual sin procesar toda la cadena de texto.

Problema 2FN / 3FN detectado

cliente, teléfono y dirección se repiten para cada pedido (Doña María aparece dos veces con los mismos datos). Además, vendedor y barrio_ruta se repiten asociados al mismo vendedor, generando dependencias parciales y transitivas.

Proceso

Formas Normales

Aplicamos las tres formas normales a la tabla de pedidos de BrunosFruit Cartagena para eliminar redundancias, garantizar la integridad de los datos y facilitar las consultas sobre clientes, productos, vendedores y rutas de reparto.

1°

Primera forma

La Primera Forma Normal (1FN) es el paso fundamental y obligatorio para comenzar a organizar una base de datos relacional de manera eficiente. Su objetivo principal es eliminar la redundancia y evitar que la base de datos se vuelva un caos difícil de consultar.
Para que una tabla cumpla con la 1FN, debe cumplir estrictamente con tres reglas:

En nuestro caso, la tabla original viola la 1FN porque las columnas productos y cantidades contienen varios valores separados por coma. Para resolverlo, descomponemos cada pedido en tantas filas como productos tenga, de modo que cada celda contenga un solo valor atómico.

1

Los atributos deben ser atómicos (Valores Únicos)

Hay multiples datos en la columna productos y cantidades

2
No deben existir grupos repetitivos

Cada producto de un pedido pasa a ser una fila independiente, eliminando los grupos repetitivos de las columnas productos y cantidades.

3
Cada fila debe ser única (Clave Primaria)

La clave primaria pasa a ser compuesta: id_pedido + id_producto. Así cada fila se identifica de forma única y podemos tener varios productos por pedido sin duplicar información.

Logramos que cada celda contenga un solo valor atómico y cada fila sea única gracias a la clave compuesta (id_pedido + id_producto). Sin embargo, aún hay datos repetidos: el cliente, el vendedor y el barrio aparecen una y otra vez en varias filas. Eso lo resolveremos en la 2FN eliminando las dependencias parciales.

2°
Segunda Forma Normal

La tabla debe estar en 1FN y cada atributo no clave debe depender de toda la clave primaria, no solo de una parte.

La 2FN elimina las dependencias parciales. En nuestra tabla 1FN, la clave es compuesta (id_pedido + producto), pero columnas como cliente, teléfono, dirección, vendedor, barrio_ruta y total solo dependen de id_pedido, no del producto. Las separamos en dos tablas.

Dependencias parciales en su tabla

En la tabla 1FN con clave compuesta (id_pedido + producto):
→ cliente, teléfono, dirección, vendedor, barrio_ruta, total dependen solo de id_pedido, no del producto.
→ cantidad es el único atributo que depende de la clave completa (id_pedido + producto).

3°
Tercera Forma Normal
La tabla debe estar en 2FN y ningún atributo no clave debe depender transitivamente de la clave primaria.

La 3FN elimina las dependencias transitivas. En nuestra tabla PEDIDOS de 2FN, teléfono y dirección dependen del cliente, no directamente del pedido. Y barrio_ruta depende del vendedor. Separamos creando las tablas CLIENTE y VENDEDOR.

Dependencias transitivas detectadas

En la tabla PEDIDOS (2FN):
→ id_pedido → cliente → teléfono, dirección
  (teléfono y dirección dependen del cliente, no del pedido)
→ id_pedido → vendedor → barrio_ruta
  (el barrio depende del vendedor asignado, no del pedido directamente)

PostgreSQL · Modelo Final 3FN
CREATE TABLE CLIENTE (
  id_cliente   SERIAL       PRIMARY KEY,
  nombre       VARCHAR(60)   NOT NULL,
  telefono     VARCHAR(15)   NOT NULL,
  direccion    VARCHAR(100)  NOT NULL
);

CREATE TABLE VENDEDOR (
  id_vendedor  SERIAL       PRIMARY KEY,
  nombre       VARCHAR(40)   NOT NULL,
  barrio_ruta  VARCHAR(60)   NOT NULL
);

CREATE TABLE PRODUCTO (
  id_producto     SERIAL       PRIMARY KEY,
  nombre          VARCHAR(40)   NOT NULL,
  precio_unitario NUMERIC(10,2) NOT NULL CHECK(precio_unitario > 0)
);

CREATE TABLE PEDIDOS (
  id_pedido    SERIAL       PRIMARY KEY,
  id_cliente   INTEGER      NOT NULL REFERENCES CLIENTE(id_cliente),
  id_vendedor  INTEGER      NOT NULL REFERENCES VENDEDOR(id_vendedor),
  total        NUMERIC(10,2) NOT NULL
);

CREATE TABLE DETALLE_PEDIDO (
  id_detalle   SERIAL       PRIMARY KEY,
  id_pedido    INTEGER      NOT NULL REFERENCES PEDIDOS(id_pedido),
  id_producto  INTEGER      NOT NULL REFERENCES PRODUCTO(id_producto),
  cantidad     INTEGER      NOT NULL CHECK(cantidad > 0),
  UNIQUE (id_pedido, id_producto)
);

Diccionario de Datos

El diccionario de datos documenta cada campo de cada tabla del modelo final 3FN: su tipo, tamaño, restricciones y significado. Sirve como guía para desarrolladores y administradores de la base de datos.

Definición

Es un catálogo estructurado que describe todos los elementos de datos del esquema, incluyendo nombre, tipo, longitud, valores permitidos, clave primaria/foránea y una descripción funcional. Facilita el mantenimiento, la integración y la comunicación entre el equipo técnico y de negocio.
